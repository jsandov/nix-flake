# Flake skeleton for evaluation-only CI (no real hardware)

Session notes from implementing ARCH-01 (bootstrap flake skeleton for the compliance-mapped NixOS AI server). Captures the patterns and gotchas encountered so they can be distilled into wiki articles on a future compile pass.

## Goal

Scaffold `flake.nix`, `hosts/ai-server/`, and the six module directories (`stig-baseline`, `gpu-node`, `lan-only-network`, `audit-and-aide`, `agent-sandbox`, `ai-services`) so that:

1. `nix flake check` succeeds against the skeleton.
2. `nix eval .#nixosConfigurations.ai-server.config.system.build.toplevel.drvPath` returns a derivation path without realising anything.
3. CI can run steps 1–2 on a stock GitHub runner that has no real disks, no hardware profile, and no access to the target server.

The pattern below lets a NixOS-targeting flake be validated purely by its evaluator, deferring any actual system build to a deployment host.

## Minimum viable options for a `nixosConfiguration` to evaluate

Even without building, the module system insists on a handful of grounded values. Leaving any of them unset aborts evaluation with errors like "The option `fileSystems.\"/\"` is used but not defined." Concretely:

| Option | Skeleton value | Why |
|---|---|---|
| `system.stateVersion` | `"24.11"` | Warns on `nixosSystem` without it; some modules read it. |
| `networking.hostName` | `"ai-server"` | Referenced transitively by several modules. |
| `fileSystems."/"` | `{ device = "/dev/disk/by-label/nixos"; fsType = "ext4"; }` | Any device string works as long as evaluation doesn't try to mount. |
| `boot.loader.systemd-boot.enable` | `true` (via `lib.mkDefault`) | `nixosSystem` needs *some* bootloader config. `systemd-boot` is cheap. `lib.mkDefault` lets ARCH-09 swap to lanzaboote without a conflict. |
| `boot.loader.efi.canTouchEfiVariables` | `true` (via `lib.mkDefault`) | Paired with systemd-boot. |
| `nixpkgs.hostPlatform` | `"x86_64-linux"` (via `lib.mkDefault`) | Avoids `nixosConfigurations` dragging platform inference through every module. |

Using `lib.mkDefault` on boot and hostPlatform options is critical — it lets the real hardware module in ARCH-09/deployment time override without the module system throwing a conflict.

There is no `--eval-only` flag or `allowTest = true` escape hatch. The minimum set above is the cost of entry.

## `checks.<system>.eval` as a CI hook

```nix
checks.${system} = {
  eval =
    (self.nixosConfigurations.ai-server.config.system.build.toplevel.drvPath or
      (throw "ai-server toplevel failed to evaluate"));
};
```

Exposing the toplevel derivation path as a `checks` output means `nix flake check` walks the entire module tree during evaluation. If anything upstream is broken — missing option, typo, bad assertion — the check fails loudly. Because the value is a *path* and not the realised output, nothing is built. This is the idiom nixpkgs CI uses.

`throw` inside the `or` clause gives a human-readable error instead of the default attribute-not-found message.

## Module stub conventions (safe no-ops)

Any of these is a valid NixOS module and contributes nothing to evaluation cost:

```nix
# Fully empty
{ }

# Convention with unused arg binder — lets future edits add `config`, `lib`, `pkgs`
# without diff noise
{ ... }:
{ }

# Commented-only stub — used for all six module directories in this repo
{ ... }:
{
  # Ownership comment explaining which control families this module will
  # implement and which TODO IDs drive the work. No options, no config.
}
```

The ownership-comment form is worth keeping as a convention: it documents intent at the module's entry point, survives `git grep` for control mapping, and doesn't impose any structural commitment before ARCH-02 (canonical config) lands.

## Input pinning choice: branch vs SHA

For the skeleton PR, `nixpkgs.url = "github:NixOS/nixpkgs/nixos-24.11"` pins to the branch tip. A committed `flake.lock` would resolve that to a concrete SHA on generation, giving reproducibility. Because the implementation environment for this PR has no `nix` CLI, the lock is generated by CI on first run.

Follow-up: once CI produces a valid lock, commit it back. Then drop the "generate if missing" step in the workflow. The lock becomes the primary reproducibility contract; the branch URL is only a hint to the upgrade tool.

## Bootstrapping `flake.lock` without a local nix binary

Three options evaluated:

1. **Hand-craft `flake.lock`.** Requires computing `narHash` values — not possible without a nix evaluator. Rejected.
2. **Commit no lock; CI runs `nix flake lock` if missing.** Gives us a starting point; each CI run may produce a slightly different lock until the first is committed. Used for this PR.
3. **Have CI upload the generated lock as an artifact and open a follow-up PR.** Cleanest long-term but adds workflow complexity for a one-shot bootstrap. Deferred.

## Gotchas encountered

- **`lib.mkDefault` is not optional.** Without it, future modules that set `boot.loader.*` or `nixpkgs.hostPlatform` at a different priority will fail with option-conflict errors. Every host-level scaffold value should carry `lib.mkDefault` by default.
- **`hosts/ai-server/hardware-configuration.nix` is a placeholder.** It sets enough for evaluation to succeed. A real deployment will replace this file with one generated by `nixos-generate-config` on the target. Keep it under `hosts/ai-server/` so the substitution is a file swap, not a module graph rewrite.
- **`timeZone = "UTC"`** in the host module is a real choice, not a stub. Compliance frameworks (NIST AU-8, PCI 10.6) expect UTC for cross-system log correlation. Set it at the host level so ARCH-10 evidence generation can rely on it.
